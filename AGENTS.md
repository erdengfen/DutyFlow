# AGENTS.md

## 1. 文档定位

本文件是本项目在 Codex 内使用的高层约束文档，用于约束项目目标、技术路线、Demo 期实现边界、系统职责划分与非目标范围。

本文件**只讨论产品与系统层面的约束**，不包含以下内容：

- 任何代码级实现细节
- 任何数据结构、表结构、字段设计
- 任何函数签名、类设计、目录级源码约束
- 任何可直接转译为代码的伪代码流程

这些内容应在后续独立的代码约束文档中定义。

---

## 2. 项目名称与一句话定义

**项目名称：** 权限感知型飞书办公 Agent（本地单用户 Demo）

**一句话定义：**
一个面向办公协作场景的权限感知型 Agent。它基于联系人关系、信息来源、事项类型和责任归属，对飞书中的消息与文件进行分层判断，只把真正需要用户介入的事项，在正确时机、以正确方式反馈给用户，并在授权边界内协助处理。

---

## 3. 项目目标

### 3.1 Demo 期核心目标

Demo 期的目标不是构建通用 Agent 平台，也不是复刻 OpenClaw 的全部能力，而是验证以下核心命题：

1. 在办公场景中，Agent 的价值不在于“会不会做很多事”，而在于“是否懂权限、责任、来源和优先级”。
2. 同样一条消息或文件，不同联系人、不同来源、不同责任关系下，Agent 的反馈方式应当不同。
3. Agent 必须在“提醒价值”和“越权风险”之间做平衡，优先体现谨慎、可解释、可审批、可留痕。
4. Demo 必须形成最小闭环：
   - 接收飞书事件
   - 判断事项权重
   - 保留必要上下文
   - 触发审批或提醒
   - 形成任务状态
   - 将结果反馈给用户

### 3.2 项目差异化目标

本项目区别于泛用型 Agent 的关键，不是工具数量、RAG 范围或复杂推理能力，而是以下两层：

- **身份层**：联系人关系、来源角色、组织语境、责任上下文
- **权重层**：重要性、紧急性、责任性、打断必要性、可执行性

项目必须始终围绕这两层展开，不允许在 Demo 期偏移为“泛用聊天助手”或“自动化工具总线”。

## 4. 总体产品边界

### 4.1 产品形态

本项目是一个：

- **本地运行**的单用户 Agent
- **飞书作为主要用户前端**
- **CLI 作为开发调试与观察台**
- **文件系统作为主要持久化媒介**

### 4.2 项目不是以下产品

本项目在 Demo 期**不是**：

- 通用对话助手
- 多用户协作平台
- 企业级 SaaS 服务
- 全功能 Agent 框架
- 自动化执行优先的“万能办公机器人”
- 大规模知识库或搜索系统
- 多 Agent 协同系统

### 4.3 交互主线

用户主要通过飞书接收以下内容：

- 重要事项提醒
- 摘要与压缩结果
- 待确认的审批请求
- 任务状态变更结果

CLI 的定位是：

- 本地调试入口
- 运行状态观察台
- 日志与事件查看入口
- 人工检查辅助界面

CLI 不是 Demo 期的主要产品前端。

## 5. 技术路线（高层）

### 5.1 总体路线

项目采用以下技术路线：

- **运行形态：** 本地单进程常驻运行
- **并发模型：** 进程内异步任务调度
- **语言：** Python
- **Agent 方案：** 自研轻量 Agent 基座，不使用通用 Agent 框架作为核心依赖
- **飞书接入：** 通过飞书开放平台能力接入事件、读取用户可授权资源、并以应用身份回馈用户
- **持久化：** 本地文件存储，优先可读、可审计、即开即用
- **安全策略：** 审批优先、读写分离、最小本地权限、最小自动执行

### 5.2 核心设计原则

1. **先闭环，再扩展**
2. **先规则，再智能**
3. **先可解释，再复杂化**
4. **先审批，再自动化**
5. **先文件存储，再考虑外部基础设施**
6. **先单用户稳定，再讨论平台化**

---

## 6. 飞书接入原则（高层）

### 6.1 接入目标

飞书是本项目的主要工作环境。Demo 期中，Agent 必须具备基于飞书进行信息接收、理解、提醒与反馈的能力。

### 6.2 身份分离原则

飞书侧的“读取”和“回写”应在设计上分离理解：

- **读取面**：尽量站在用户视角读取用户已授权且有权访问的资源
- **回馈面**：由独立的 Agent/应用身份向用户发送提醒、审批、结果与状态更新

### 6.3 Demo 期接入目标

Demo 期只要求建立以下能力闭环：

- 接收消息相关事件
- 接收文件/文档相关线索
- 在权限范围内读取必要上下文
- 将提醒、审批与结果发送回用户

### 6.4 Demo 期不追求

- 覆盖飞书全部 API
- 覆盖用户“理论上能看到的一切”资源
- 实现复杂的跨资源联动自动化
- 实现企业级权限治理体系

飞书接入层的目标是**支撑核心判断逻辑**，而不是成为 Demo 的亮点本身。

---

## 7. 本地运行与权限原则

### 7.1 本地运行定位

本项目是本地单用户 Agent，默认运行在用户个人环境中。它不是云端托管服务，也不要求数据库、容器编排或多服务部署。

### 7.2 本地权限原则

Demo 期遵循**最小本地权限原则**：

允许：

- 网络访问
- 本地项目目录读写
- 本地配置与日志落盘
- 本地常驻运行
- 必要的浏览器授权流程配合

默认不引入：

- 整机文件系统完全访问
- 本地命令高危自动执行
- 桌面控制
- 浏览器自动控制
- 麦克风、摄像头、屏幕等额外本地权限

### 7.3 可见性原则

除敏感密钥信息外，Demo 期的重要产出应尽量满足：

- 用户可查看
- 用户可理解
- 用户可追溯
- 用户可核对

本项目不应把关键状态隐藏在不可见的内部黑盒中。

---

## 8. 持久化路线（高层）

### 8.1 总体原则

Demo 期采用**文件优先**的本地持久化路线。

### 8.2 为什么采用文件存储

采用文件存储而非数据库的原因：

- 即开即用
- 便于单用户本地部署
- 便于调试与人工检查
- 便于形成审计轨迹
- 降低环境搭建复杂度
- 与项目“安全、可见、可控”的定位一致

### 8.3 文件存储承载范围

文件存储只需要承载以下高层对象：

- 运行日志
- 事件记录
- 事项状态
- 审批状态
- 任务清单
- 上下文压缩结果
- 汇总报告

### 8.4 不在本文件中定义

本文件**不定义**具体的数据文件格式、字段、分层目录结构或命名方案。这些内容必须在独立的代码约束文档中确定。

## 9. Demo 期必须实现的核心基架

以下模块构成 Demo 期的系统最小闭环。Codex 在任何技术实现上都必须以这几部分为中轴，不得偏离。

### 9.1 事件入口层

职责：

- 接收飞书事件
- 将输入统一视为“待判断事项”的来源
- 支撑消息、文件、文档等事件线索进入主流程

要求：

- 能持续接收事件
- 能将事件传入内部流程
- 能作为整个系统的起点

### 9.2 身份与来源存储层

职责：

- 维护与事项判断相关的高层身份信息与来源信息
- 为每个事件补充其“来自谁、来自哪里、在何种关系下出现”的最小语境

说明：

这一层是项目的核心差异化来源之一。它不是辅助模块，而是主判断链路的一部分。

### 9.3 事件权重决策层

职责：

- 根据联系人关系、信息来源、事项类型和责任归属，对事项进行分层判断
- 决定一条事项是应该即时提醒、进入摘要、等待审批，还是暂不打断用户

说明：

这一层必须是 Demo 的重点模块。无论内部采用规则、半结构化策略或其它方式，最终目标都不是“分类准确率”本身，而是证明 Agent 的行为会因为身份层与责任层而发生差异。

### 9.4 简单上下文保存与压缩层

职责：

- 保留必要的近场上下文
- 避免事项判断完全脱离上下文
- 对近期上下文做轻量压缩，供后续提醒或审批使用

边界：

- 只做简单、近场、必要的上下文支持
- 不将其扩展为完整记忆系统
- 不将其扩展为大范围检索系统

### 9.5 人机交互的权限核实与审批接口

职责：

- 对敏感动作进行确认
- 保证 Agent 的行为符合“审批优先”的原则
- 使用户能够明确知道 Agent 想做什么、为何要做、是否允许继续

要求：

- 审批流程必须可见
- 审批结果必须可追踪
- 高风险动作不得静默执行

### 9.6 用户可查看的任务清单与状态层

职责：

- 将已识别事项沉淀为用户可见的任务/状态对象
- 呈现事项当前状态，而非只做一次性提醒

要求：

- 用户必须能够查看当前有哪些事项被接住了
- 用户必须能够理解这些事项目前处于何种阶段

### 9.7 用户回馈层

职责：

- 将系统判断结果反馈给用户
- 将提醒、摘要、审批、状态更新等内容通过飞书返回

说明：

此层是 Demo 可见度最高的模块之一。其价值不在于形式丰富，而在于体现“正确时机、正确方式、正确粒度”的反馈能力。

## 10. Demo 期明确不实现的部分

以下内容在 Demo 期只允许预留方向或接口概念，不允许作为主要建设目标。

### 10.1 长期向量记忆库

不在 Demo 期实现完整长期记忆或向量记忆能力。

允许：

- 预留高层扩展方向
- 预留未来接入位置

不允许：

- 将大量时间投入在向量库选型、嵌入检索、召回策略、记忆融合等工作上

### 10.2 完整联系人画像系统

不在 Demo 期实现复杂、长期演化的完整联系人画像系统。

允许：

- 使用最小必要的人物关系抽象
- 使用演示用占位信息或有限样本关系

不允许：

- 把项目重心转移到画像建模本身

### 10.3 RAG 知识库

不在 Demo 期实现完整的个人知识库或网络知识库增强系统。

允许：

- 预留高层扩展点
- 保持与未来知识增强能力的兼容性

不允许：

- 把检索、分片、重排、多源融合等能力当作 Demo 核心

### 10.4 外部 MCP 工具生态

不在 Demo 期重点建设外部 MCP 工具生态。

允许：

- 未来可扩展的能力预留

不允许：

- 为了扩展性而牺牲 Demo 闭环
- 为了接更多工具而偏离主线

### 10.5 复杂 Agent Loop

不在 Demo 期实现以下复杂能力：

- 多 Agent 协作
- 自反思/自纠错框架
- 长任务自动拆解
- 多轮规划维护
- 通用自治执行回路

说明：

Demo 期必须坚持“流程稳定优先于智能复杂度”。

## 12. 核心流程约束（高层）

虽然本文件不定义代码流程，但项目必须坚持以下高层行为顺序：

1. 先接住事件
2. 再补全来源与身份语境
3. 再做权重判断
4. 再决定是否需要上下文支持
5. 再决定是否需要审批
6. 再更新任务状态
7. 最后才向用户反馈

不允许出现以下偏移：

- 先自动执行，再补审批
- 先把复杂知识检索接上，再做权重判断
- 先做大而全的记忆系统，再完成事项闭环
- 先做对话体验，再忽略任务状态与审计能力

---

## 13. 产品安全与可信原则

### 13.1 审批优先

所有可能改变外部状态、代表用户表达立场、或带来误操作风险的行为，必须优先进入审批或确认语境。

### 13.2 留痕优先

关键判断和关键动作必须可追溯。Demo 期虽然不追求完整审计系统，但必须保留“发生过什么”的最低可核对能力。

### 13.3 解释优先

系统反馈应尽量让用户理解：

- 为什么它被认为重要
- 为什么现在要提醒
- 为什么需要审批

### 13.4 读写分离

项目在理念上坚持：

- 读取尽量站在用户已授权视角
- 回馈尽量通过独立 Agent 身份完成

### 13.5 可见优先于自动化

在 Demo 期，如果“自动做很多事”和“让用户清楚知道系统在做什么”发生冲突，应优先选择后者。

---

## 14. 开发优先级约束

Codex 在推进实现时，必须遵守以下优先级顺序：

### 第一优先级

- 事件入口打通
- 核心判断链路形成
- 飞书回馈链路可用
- 任务状态可见
- 审批能力可演示

### 第二优先级

- 简单上下文支持
- 更清晰的身份层表达
- 更稳健的状态流转
- 更好的可观察性与日志可读性

### 第三优先级

- 预留未来扩展点
- 组织更好的模块边界
- 为后续 RAG / 记忆 / 工具扩展留接口概念

### 不应抢占优先级的事项

- 大规模工具接入
- 长期记忆系统
- 多 Agent 系统
- 高复杂规划系统
- 泛化知识库能力

---

## 15. Demo 验收标准（产品层）

Demo 期的完成标准不是“模块名义上存在”，而是至少能演示以下事实：

1. 能接收到飞书侧的真实输入
2. 能依据联系人/来源/责任差异，做出不同反馈
3. 能对重要事项进行及时提醒
4. 能对敏感动作触发审批/确认
5. 能把事项沉淀为可查看的任务或状态
6. 能将结果回传给用户
7. 能以文件形式保留关键运行产物，便于检查和解释

只要以上 7 点成立，Demo 即可视为闭环成立。

## 16. 后续演化方向（仅方向，不进入 Demo）

以下内容只能作为后续演进方向，不得在 Demo 期喧宾夺主：

- 长期记忆和失败经验沉淀
- 更完整的联系人画像
- 私域知识库与网络知识增强
- 外部工具和技能生态
- 复杂任务拆解与规划
- 多 Agent 协同
- 更强的组织权限治理能力

这些能力必须建立在 Demo 核心闭环稳定之后再讨论。

## 17. 给 Codex 的执行性约束

Codex 在参与本项目时，必须始终遵守以下原则：

1. 不将项目重心偏移到泛用 Agent 能力展示
2. 不主动引入数据库、消息队列、容器编排或多服务拆分
3. 不主动引入大型 Agent 框架作为主骨架
4. 不将长期记忆、RAG、外部工具生态视为 Demo 主任务
5. 不在没有明确需求的情况下扩大自动执行能力
6. 不在本文件范围内设计数据结构、表结构或代码级协议
7. 所有实现建议都应优先服务于“身份层 + 权重层 + 审批流 + 任务可见性”这条主线

---

## 18. 最终判断

本项目在 Demo 期必须被实现为：

> 一个本地运行、飞书作为前端、文件可审计、审批优先、以身份层与权重层为核心差异化的办公协作权限感知型 Agent。

任何偏离上述定义的实现方向，都应被视为范围漂移。
