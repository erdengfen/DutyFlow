# CODE_STYLE.md

本文档记录项目代码规范。凡涉及代码编写、修改或重构，必须先阅读并遵守本文档。

## 1. 规模限制

- 单个代码文件不得超过 800 行。
- 单个方法或函数不得超过 50 行。
- 嵌套调用或嵌套控制结构不得超过 3 层。
- 单个方法或函数内的主要条件分支不得超过 3 个。

如确实接近限制，应优先拆分职责、提取小函数或调整模块边界，不得用压缩写法规避行数限制。

## 2. 职责与注释

- 每个文件头部必须使用清晰中文注释说明该文件职责。
- 每个类必须使用中文注释说明职责和适用边界。
- 每个方法或函数必须使用中文注释说明用途。
- 注释应解释“为什么存在”和“负责什么”，不得堆砌显而易见的代码翻译。

## 3. 命名与结构

- 命名必须直接表达业务含义，避免模糊缩写和泛化名称。
- 一个文件只承担一个主要职责。
- 一个方法或函数只完成一个明确动作。
- 复杂判断应拆分为具名函数，使调用处能直接读出业务意图。
- 公共逻辑可以提取复用，但不得为了抽象而引入无实际收益的层级。

## 4. 配置与密钥

- 一切 `api_key`、密钥、令牌、认证信息都必须来自统一入口 `.env`。
- 所有用户配置也必须来自 `.env`，包括飞书接入层所需的飞书账号、认证信息、应用配置和授权相关配置。
- 代码中禁止硬编码密钥、账号、认证参数或用户私有配置。
- 代码中禁止为真实密钥提供默认值。
- 示例值只能出现在示例文档或环境模板中，并且必须明确是占位示例。

## 5. 错误处理

- 涉及外部 API、文件读写、配置读取、审批状态和任务状态的代码必须有明确错误处理。
- 错误信息应能帮助定位问题，但不得泄露密钥、令牌或用户隐私配置。
- 不得静默吞掉异常；确需降级时必须保留可追踪记录。

## 6. 与测试规范的关系

- 每个 `.py` 文件必须满足 `docs/TESTING.md` 中的自测入口要求。
- 新增或修改完整功能分块时，必须同步补充或更新 `test/` 下对应测试文件。
