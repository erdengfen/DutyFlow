# AGENTS.md

## 1. 项目定义

本项目是一个本地单用户运行的权限感知型飞书办公 Agent Demo。

核心目标是验证：Agent 能基于联系人关系、信息来源、事项类型和责任归属，对飞书消息、文件、文档线索进行权重判断，并以提醒、摘要、审批请求、任务状态更新等形式反馈给用户。

项目核心差异化只聚焦两层：

- 身份层：联系人关系、来源角色、组织语境、责任上下文
- 权重层：重要性、紧急性、责任性、打断必要性、可执行性

Demo 期不建设通用聊天助手、企业级 SaaS、多用户协作平台、完整 Agent 框架、大规模知识库、多 Agent 系统或自动化工具总线。

## 3. 技术路线

- 运行形态：本地单进程常驻运行
- 并发模型：进程内异步任务调度
- Agent 方案：自研 Agent 基座，不使用大型通用 Agent 框架作为主骨架
- 飞书接入：通过飞书开放平台接收事件、读取授权范围内必要上下文，并以应用/Agent 身份回馈用户
- 持久化：本地文件存储，服务于日志、事件、任务、审批、上下文摘要和报告的可查看、可追溯需求
- 交互入口：飞书作为主要用户前端，CLI 作为本地调试、观察和人工检查入口
- 安全策略：审批优先、读写分离、最小本地权限、最小自动执行

## 4. 技术选型

- 开发语言：Python
- 环境管理：uv
- Agent 基座：自研实现
- 事件与回馈入口：飞书开放平台
- 用户前端：飞书
- 开发调试入口：CLI
- 持久化介质：本地文件系统
- 部署形态：本地单用户运行

Demo 期默认允许：

- 网络访问
- 本地项目目录读写
- 本地配置与日志落盘
- 本地常驻运行
- 必要的浏览器授权流程配合

Demo 期默认不引入：

- 数据库、消息队列、容器编排或多服务拆分
- 整机文件系统完全访问
- 本地高危命令自动执行
- 桌面控制、浏览器自动控制
- 麦克风、摄像头、屏幕等额外本地权限

## 5. 项目架构

Demo 期系统由以下模块组成，模块职责按主流程组织。

```text
Feishu 工作环境
  消息 / 文件 / 文档 / 会话 / 用户交互
        |
        v
事件入口层
  接收飞书事件，将消息、文件、文档等输入统一送入本地主流程
        |
        v
身份与来源补全层
  补充联系人关系、来源类型、事项类型、责任语境
        |
        v
事件权重决策层
  判断即时提醒、进入摘要、等待审批、形成任务或暂不打断
        |
        +---------------------+
        |                     |
        v                     v
上下文保存与压缩层       权限核实与审批层
  保存近场上下文          对敏感动作发起可见确认
  生成轻量摘要            记录审批结果
        |                     |
        +----------+----------+
                   |
                   v
任务清单与状态层
  将已识别事项沉淀为用户可查看的任务和处理状态
                   |
                   v
用户回馈层
  通过飞书返回提醒、摘要、审批请求、状态更新
                   |
                   v
本地文件持久化与审计层
  保留事件、日志、上下文摘要、审批、任务、报告等关键产物
```

主流程顺序必须保持：

1. 接收事件
2. 补全来源与身份语境
3. 做权重判断
4. 按需保存或压缩上下文
5. 按需触发审批
6. 更新任务状态
7. 向用户反馈
8. 落盘关键产物

## 6. Demo 期实现范围

Demo 期必须实现以下闭环能力：

- 接收飞书侧真实输入，包括消息事件，以及文件或文档相关线索
- 在授权范围内读取必要上下文
- 维护最小身份与来源信息，用于区分联系人关系、来源角色和责任语境
- 基于身份层与权重层对事项做分层判断
- 对重要事项及时提醒，对低打断事项进入摘要或状态沉淀
- 对敏感动作发起审批或确认，审批流程和结果必须可见、可追踪
- 将已识别事项沉淀为用户可查看的任务或状态对象
- 通过飞书向用户返回提醒、摘要、审批请求、任务状态变更结果
- 以本地文件形式保留关键运行产物，支持人工检查和审计
- 提供 CLI 能力用于本地调试、运行状态观察、日志与事件查看

Demo 期可以预留但不作为建设重点：

- 长期记忆
- 完整联系人画像
- RAG 或个人知识库
- 外部 MCP 工具生态
- 复杂任务拆解与规划
- 多 Agent 协同
- 更完整的组织权限治理

Demo 期不实现：

- 完整长期向量记忆库
- 复杂联系人画像系统
- 大规模知识库检索、分片、重排、多源融合
- 通用自治执行回路
- 多用户平台化能力
- 企业级权限治理体系

## 7. 执行约束

Codex 在本项目中必须遵守以下约束：

- 优先完成事件入口、核心判断链路、飞书回馈链路、任务状态可见性和审批能力
- 所有实现建议必须服务于“身份层 + 权重层 + 审批流 + 任务可见性”
- 先规则，再智能；先闭环，再扩展；先可解释，再复杂化
- 不主动引入数据库、消息队列、容器编排、多服务拆分或大型 Agent 框架
- 不将长期记忆、RAG、外部工具生态、多 Agent 系统作为 Demo 主任务
- 不在没有明确需求的情况下扩大自动执行能力
- 不允许先自动执行再补审批
- 不允许先建设复杂知识检索、长期记忆或通用对话体验，再补核心事项闭环
- 关键判断、关键动作、审批结果和任务状态必须可查看、可解释、可追溯
- 对关键开关、阈值、上限类数值必须在代码旁用中文注释说明用途，如最大并发数、最大轮数、超时秒数、重试次数；禁止保留无说明的 magic number
- 涉及改变外部状态、代表用户表达立场、或存在误操作风险的动作，必须先进入审批或确认
- 飞书读取面应站在用户授权范围内，回馈面应通过独立应用/Agent 身份完成
- 本文件不定义数据结构、表结构、字段设计、函数签名、类设计、目录结构或可直接转译为代码的伪代码流程

## 8. 验收标准

Demo 闭环成立必须同时满足：

1. 能接收到飞书侧真实输入
2. 能依据联系人、来源、责任差异做出不同反馈
3. 能对重要事项进行及时提醒
4. 能对敏感动作触发审批或确认
5. 能把事项沉淀为可查看的任务或状态
6. 能将提醒、摘要、审批请求和状态结果回传给用户
7. 能以本地文件形式保留关键运行产物，便于检查和解释

只要以上标准未满足，新增长期记忆、知识库、复杂 Agent Loop、外部工具生态或平台化能力均视为范围漂移。
